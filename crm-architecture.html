§ 01 — System Architecture

Two planes, one platform.

A control plane where the super admin provisions accounts, defines roles, and toggles page access — sitting above a tenant plane where each agency's admins and employees work inside their own isolated workspace. A Next.js frontend talks to a modular backend; every domain is its own module behind a shared gateway, and all tenant data is scoped by account_id.

§ 02 — Roles & Access Model

Who can touch what.

Three tiers. The super admin owns the platform and every account on it. Inside an account, an admin runs the workspace including its staff; an employee does the same work minus the people-management page. Crucially, the page-to-role mapping is data, not code — the super admin edits it from the console.

i.

Super Admin platform

Lives above all accounts. Creates and configures accounts, switches into any account, creates the first admin per account, and edits roles, pages, fields, and feature flags globally. Not bound by account_id.

ii.

Admin account

Full access to every page in their account including Employee Management, where they create and manage employee accounts. Scoped strictly to their own account_id.

iii.

Employee account

Identical to the admin in every page except Employee Management, which is hidden and route-guarded. Same tenant scope.

Access is resolved, never hardcoded.

On login the gateway computes an effective permission set = role defaults + per-account overrides + feature flags, then caches it in Redis. The frontend reads the same set to show/hide nav items, and the backend enforces it on every request. Changing a checkbox in the super admin console is all it takes to move a page between roles.

§ 03 — Data Model

What lives where.

A tenancy spine (Account → User) with an RBAC layer beside it, and the business objects (Contact, Pipeline → Stage → Opportunity) hanging off each account. Every business row carries account_id; dynamic data lives in custom-field definitions plus JSONB values so new fields never require a migration.

Core entities
01

Account

A tenant / agency workspace. Created by the super admin. Owns all business data and its own settings, branding, and feature flags. Type may be extended later (agency, sub-account, etc.).

02

User

A login. Has a role (super_admin / admin / employee) and — for non-super-admins — an account_id. Super admins have a null account and can act inside any account via switching.

03

Role & PagePermission

Role defaults plus an editable join of {account, role, page, can_view/edit}. This table is what the super admin edits to move pages between roles per account.

04

Contact

A person/business created manually via the form: name, emails, phones, business name, tags, contact type, timezone, channels/DND. Standard fields + custom-field values.

05

Pipeline & Stage

A named pipeline with ordered stages (New Lead, Contacted, Qualified, Proposal, Closed…). Pipelines are selectable from the dropdown on the opportunities page. Stages carry order + display config.

06

Opportunity

A deal linked to a contact, sitting on one stage of one pipeline: name, value, status (open/won/lost), owner, followers, source, business name, tags.

07

CustomFieldDefinition

Per object (Contact / Opportunity): key, label, type, folder, options. Defines the dynamic schema; values stored in JSONB on the record. Standard fields (state, city, website…) modelled the same way.

08

Tag & AuditLog

Reusable labels scoped to an account, plus an append-only audit trail of who changed what — the backbone for traceable, editable configuration.

§ 04 — Backend Module Structure (Apps)

Many modules, one boundary.

A modular monolith: a single deployable, internally split into self-contained modules ("apps") that each own their routes, services, models, and validation. Modules talk through interfaces, not each other's internals — so a new module is added, never stitched in. NestJS (TypeScript) pairs naturally with the Next.js frontend, but the structure maps cleanly onto Django apps or FastAPI routers too.

Adding functionality = adding a folder.

A new capability (say tasks or calendars) is a new module folder of the same shape, one registration line in app.module, and a page entry in page-registry. Nothing existing is edited. Tenant scoping, RBAC, and audit are inherited from core for free.

§ 06 — Frontend Architecture

Next.js, kept DRY.

App Router with route groups for each plane. A small set of generic, reusable components (table, form-builder, drawer, kanban) renders every page — Contacts and Opportunities are configurations of the same primitives, not bespoke screens. Navigation and routes are driven by the permission set, so visibility tracks the super admin's edits automatically.

DRY

One table, one form, many pages

Contacts and Opportunities both render DataTable + RecordDrawer; only their column & field config differs. New fields appear automatically because FormBuilder reads custom-field definitions.

Gate

Permission-driven UI

The nav and route guards consume /me/permissions. When the super admin moves a page between roles, the menu and access change with no redeploy. The Employee page simply isn't rendered or routable for employees.

§ 07 — Core Flows

How it moves.

The end-to-end journeys, from the super admin provisioning an account down to an employee dragging a deal across a pipeline.

01Provision

Super admin creates an account

From the console, the super admin creates a new account (tenant) and its first admin user.

  • POST /accounts creates the tenant + default pipeline/fields
  • POST /accounts/:id/users creates the admin (role = admin)
  • Default page-access rows are seeded; super admin can tweak them
  • Admin receives credentials and logs in scoped to that account
02Switch

Super admin switches accounts

To work inside any tenant, the super admin "enters" it — getting a session scoped to that account_id while retaining elevated rights.

  • POST /accounts/:id/switch returns a scoped session/JWT
  • All subsequent reads/writes filter by that account
  • Action is recorded in the audit log
  • Switching back returns to the platform-level context
03Staff

Admin creates an employee

On the admin-only Employee Management page, the admin adds staff who then share every page except this one.

  • POST /users with role = employee, same account_id
  • Employee logs in; nav is built from their permission set
  • Employee Management is absent and route-guarded for them
04Contact

Create a contact (manual form)

The "Add Contact" drawer collects the standard fields plus any custom fields; first name is required.

  • Drawer rendered by FormBuilder from field definitions
  • Client + server validation (required first name, email/phone format)
  • POST /contacts saves with account_id + custom values
  • List/Smart List refreshes via the shared DataTable
05Pipeline

Create a pipeline

From Opportunities → Pipelines, a named pipeline is built with ordered, renamable stages and display colours.

  • POST /pipelines with stages [New Lead, Contacted, …]
  • Stages carry order + "show in reports" + colour mode
  • Pipeline immediately appears in the dropdown selector
06Opportunity

Create & move an opportunity

Pick a pipeline from the dropdown, add a deal tied to a contact, then drag it across stages on the kanban.

  • POST /opportunities with contact, pipeline, stage, value
  • KanbanBoard renders stages; drag triggers PATCH /:id/stage
  • Status (open/won/lost), owner, followers, tags all editable
07Configure

Super admin edits pages, fields & access

The control plane edits the matrix, defines new custom fields, and can register entirely new pages.

  • PATCH /rbac/page-access moves a page between roles
  • POST /custom-fields adds a field — no migration needed
  • POST /page-registry registers a new page/feature
  • Permission cache invalidates; UI updates on next load
§ 08 — Modularity & Extensibility

Built to be appended to.

The whole point of the structure: today it's two pages, but every seam is designed so tomorrow's features slot in without rewrites. Four mechanisms carry that promise.

i.

Self-contained modules add a folder

Each module owns its routes, logic, data, and validation. New features arrive as new modules registered in one place — existing code is never modified.

ii.

Dynamic fields no migration

Custom-field definitions + JSONB values mean "every small thing" — a new field on contacts or opportunities — is a config change, not a schema change.

iii.

Page registry + RBAC super-admin

Pages are data. The super admin registers a page and grants it to roles per account, so access is editable from the console exactly as required.

iv.

Feature flags per account

Modules and pages can be switched on per account, enabling staged rollout and account-type differences without branching the codebase.
